# Руководство QC менеджера (Quality Manager)

> Moonjar PMS -- Система управления производством каменных изделий

---

## 1. Роль и зона ответственности

Quality Manager (QC менеджер) -- ключевая роль в обеспечении качества продукции на всех этапах производства. Роль `quality_manager` в системе.

### Основные обязанности

- **Проверка качества** позиций после обжига и на этапе сортировки
- **Фиксация дефектов** с указанием причины и фотодоказательствами
- **Блокировка производства** (QM Block) при обнаружении критических дефектов
- **Управление Problem Cards** -- карточки проблем на производстве
- **Контроль ремонтной очереди** -- отслеживание SLA по ремонту дефектных позиций
- **Анализ фото** через LLM (AI-анализ качества, весов, упаковки)
- **Взаимодействие с PM** -- уведомления о критических дефектах

### Доступ в системе

- URL дашборда: `/quality`
- Доступ: `quality_manager`, `owner`, `administrator`
- Права на создание/редактирование: причины дефектов, инспекции, QM-блоки, Problem Cards
- Привилегия `require_quality` = owner + administrator + quality_manager

## 2. Дашборд QC: обзор

При входе в систему QC менеджер видит дашборд с тремя вкладками и четырьмя KPI-карточками.

### KPI-карточки (верхняя панель)

| Метрика | Цвет | Описание |
|---------|------|----------|
| **Pending QC** | Оранжевый | Количество позиций, ожидающих проверки качества (статус `sent_to_quality_check`) |
| **Blocked** | Красный | Активные QM-блоки (неразрешённые) |
| **Problem Cards** | Жёлтый | Открытые карточки проблем (статусы `open` и `in_progress`) |
| **Today's Checks** | Зелёный | Количество инспекций, проведённых сегодня |

### Вкладки

1. **QC Queue** -- Очередь позиций на проверку
2. **QM Blocks** -- Активные блокировки производства
3. **Problem Cards** -- Карточки проблем

## 3. Проверка качества на каждом этапе

### 3.1 Этапы контроля качества (QcStage)

Система поддерживает три этапа QC-проверки:

| Этап | Код | Когда проводится |
|------|-----|------------------|
| **Глазурование** | `glazing` | После нанесения глазури, до загрузки в печь |
| **Обжиг** | `firing` | После выхода из печи (основной этап) |
| **Сортировка** | `sorting` | Финальная проверка перед упаковкой |

### 3.2 Автоматическое назначение QC-проверок

После выхода партии (batch) из печи система автоматически назначает QC-задачи:

1. Берутся все позиции в партии со статусом `fired`
2. Из таблицы `quality_assignment_config` определяется текущий процент выборки (базовый -- 2%)
3. Позиции с флагом `mandatory_qc` проверяются **всегда**
4. Из остальных отбирается случайная выборка по проценту
5. Выбранные позиции переходят в статус `sent_to_quality_check`
6. Создаются блокирующие задачи типа `QUALITY_CHECK`, назначенные на роль `quality_manager`

**Важно:** Процент выборки автоматически увеличивается при обнаружении дефектов (шаг увеличения настраивается в `quality_assignment_config`, по умолчанию +2% за каждый найденный дефект, максимум 100%).

### 3.3 Процесс инспекции

**Жизненный цикл позиции через QC:**

```
fired --> sent_to_quality_check --> [QC инспекция]
                                      |
                                      +--> OK --> quality_check_done
                                      |
                                      +--> Defect --> blocked_by_qm
```

**Порядок работы в интерфейсе (вкладка QC Queue):**

1. Выбрать позицию из очереди (карточка показывает: номер заказа, цвет, размер, количество)
2. Открывается форма инспекции
3. Нажать **OK** или **Defect**
4. Если Defect -- обязательно выбрать причину дефекта из справочника
5. Опционально: добавить заметки
6. Нажать **Submit Inspection**

### 3.4 Результаты инспекции (QcResult)

| Результат | Код | Что происходит |
|-----------|-----|----------------|
| **OK** | `ok` | Позиция переходит в статус `quality_check_done` |
| **Defect** | `defect` | Позиция блокируется (`blocked_by_qm`), автоматически создаётся QM Block |

## 4. Управление дефектами

### 4.1 Причины дефектов (Defect Causes)

Справочник причин дефектов доступен через API `/api/quality/defect-causes`. Каждая причина имеет:

- **code** -- уникальный код (например, `CRACK_01`, `GLAZE_BUBBLE`)
- **category** -- категория дефекта
- **description** -- описание

QC менеджер может **создавать новые причины** дефектов (эндпоинт POST `/api/quality/defect-causes`).

### 4.2 Этапы возникновения дефектов (DefectStage)

| Этап | Код | Описание |
|------|-----|----------|
| Входной контроль | `incoming_inspection` | Дефект сырья/материалов |
| До глазурования | `pre_glazing` | Дефект заготовки |
| После ангоба | `after_engobe` | Дефект после нанесения ангоба |
| Перед печью | `before_kiln` | Дефект обнаружен при загрузке |
| После обжига | `after_firing` | Основной -- трещины, пузыри, деформация |
| Сортировка | `sorting` | Дефект обнаружен при финальной сортировке |

### 4.3 Исходы дефектных позиций (DefectOutcome)

| Исход | Код | Описание |
|-------|-----|----------|
| Возврат в работу | `return_to_work` | Незначительный дефект, можно продолжить |
| Списание | `write_off` | Критический дефект, позиция списывается |
| Шлифовка | `grinding` | Отправляется на шлифовку |
| Ремонт | `repair` | Требуется переглазуровка |
| Повторный обжиг | `refire` | Уже заглазурована, нужен повторный обжиг |
| Переглазуровка | `reglaze` | Снятие глазури и повторное нанесение |
| На склад | `to_stock` | Пригодна для других заказов |
| На Мана | `to_mana` | Передаётся на Мана (вторичный канал) |

### 4.4 Запись фактического процента брака

Через эндпоинт POST `/api/defects/record` менеджмент фиксирует фактический процент дефектов по позиции. Если процент превышает порог -- автоматически создаётся задача для QC менеджера на анализ "5 Why" (поиск корневой причины).

### 4.5 Коэффициенты дефектов

Система использует коэффициенты дефектов для планирования запаса:
- **По типу глазури** -- учитывается специфика каждой глазури
- **По типу продукта** -- плитка, столешница, раковина, 3D

Комбинированный коэффициент = коэффициент глазури + коэффициент продукта. Он используется для расчёта `quantity_with_defect_margin` -- количества с учётом запаса на брак.

Owner/CEO могут вручную переопределить коэффициент для конкретной позиции.

## 5. QM-блоки (блокировка производства)

### 5.1 Что такое QM Block

QM Block -- это механизм блокировки позиции (или партии) QC менеджером при обнаружении критического дефекта. Блокировка **останавливает** движение позиции по производственному процессу до разрешения проблемы.

### 5.2 Типы блоков (QmBlockType)

- **position** -- блокировка конкретной позиции
- **batch** -- блокировка целой партии
- **kiln** -- блокировка печи
- **process** -- блокировка технологического процесса

### 5.3 Создание блока

Блок создаётся автоматически при результате инспекции "Defect", либо вручную через интерфейс / API.

При создании блока:
1. Позиция переходит в статус `blocked_by_qm`
2. Предыдущий статус сохраняется в метаданных задачи
3. Создаётся блокирующая задача для Production Manager (приоритет 4 -- высокий)
4. PM и CEO получают push-уведомления

### 5.4 Разрешение блока (вкладка QM Blocks)

1. Найти блок в списке активных блоков (отображаются: причина, severity, тип, дата)
2. Нажать **Resolve**
3. Ввести заметку о разрешении (обязательно)
4. Подтвердить в диалоге

При разрешении:
- Позиция возвращается к предыдущему статусу (обычно `fired`)
- Все связанные блокирующие задачи закрываются
- PM получает уведомление о разблокировке

### 5.5 Уровни severity

| Уровень | Цвет | Описание |
|---------|------|----------|
| `critical` | Красный | Немедленное внимание, полная остановка |
| `high` | Оранжевый | Высокий приоритет |
| `medium` | Жёлтый | Средний приоритет |
| `low` | Серый | Низкий приоритет |

## 6. Сортировка и сплит позиций

После обжига позиция переходит в статус `transferred_to_sorting`. На этапе сортировки определяется качество каждого изделия и позиция разделяется (split) на категории:

| Категория | Что дальше |
|-----------|-----------|
| **good** (годные) | Родительская позиция -> `packed` (+ автоматическое списание упаковки) |
| **refire** (повторный обжиг) | Дочерняя позиция -> `refire`, возвращается в очередь на печь |
| **repair** (ремонт) | Дочерняя позиция -> `sent_to_glazing`, попадает в ремонтную очередь с SLA |
| **color_mismatch** (несовпадение цвета) | Дочерняя позиция -> `planned`, начинает путь заново |
| **grinding** (шлифовка) | Запись в GrindingStock |
| **write_off** (списание) | Запись DefectRecord, количество списывается |

**Правило:** сумма всех категорий должна точно равняться количеству в позиции.

### Ремонтная очередь

Позиции, отправленные на ремонт, попадают в `repair_queue` с отслеживанием SLA. Эндпоинт GET `/api/defects/repair-queue?factory_id=...` возвращает:
- Список позиций в ремонте
- Статус SLA (в норме / просрочен)
- Общее количество просроченных

## 7. Problem Cards (Карточки проблем)

Problem Card -- инструмент фиксации любых производственных проблем, не обязательно связанных с конкретной позицией.

### Поля карточки

| Поле | Обязательное | Описание |
|------|:---:|----------|
| factory_id | Да | Фабрика (выбирается автоматически) |
| description | Да | Описание проблемы |
| location | Нет | Место обнаружения (цех, участок) |
| actions | Нет | Предпринятые действия |
| status | Да | Статус карточки |

### Жизненный цикл

```
open --> in_progress --> resolved / closed
```

### Работа в интерфейсе (вкладка Problem Cards)

1. Нажать **New Problem Card**
2. Заполнить описание (обязательно) и место (опционально)
3. Нажать **Submit**
4. Для закрытия -- нажать **Close** на карточке и подтвердить

## 8. Фотоанализ через AI

QC менеджер может использовать AI (Claude Vision) для анализа фотографий:

**Эндпоинт:** POST `/api/quality/analyze-photo`

### Типы анализа

| Тип | Код | Что анализирует |
|-----|-----|-----------------|
| Качество | `quality` | Обнаружение дефектов: трещины, проблемы с цветом, неровности |
| Весы | `scale` | Считывание веса, определение цвета пигмента |
| Упаковка | `packing` | Считывание этикетки: номер заказа, количество, размер |

### Результат анализа

- **readings** -- считанные значения
- **confidence** -- уровень уверенности
- **issues** -- обнаруженные проблемы
- **raw_description** -- текстовое описание

Можно привязать анализ к конкретной позиции через параметр `position_id` для дополнительного контекста.

### Фото инспекции

К каждой инспекции можно прикрепить фотографии через POST `/api/quality/inspections/{id}/photo` (максимум 5 МБ на фото, только изображения).

## 10. KPI качества

### Основные метрики

| KPI | Источник | Описание |
|-----|----------|----------|
| **Pending QC** | `/api/quality/stats` | Позиции в очереди на проверку |
| **Blocked** | `/api/quality/stats` | Активные QM-блоки |
| **Open Problem Cards** | `/api/quality/stats` | Открытые карточки проблем |
| **Inspections Today** | `/api/quality/stats` | Проверки за сегодня |
| **Defect Rate** | `/api/defects/supplier-reports` | Процент брака по поставщику/глазури |
| **Repair Queue Overdue** | `/api/defects/repair-queue` | Просроченные ремонты |
| **Defect Spike** | `/api/analytics/anomalies` | Аномальный рост дефектов (автоматическое обнаружение) |

### Адаптивный процент выборки

Система автоматически адаптирует процент QC-выборки:

- **Базовый**: 2% (настраивается в `quality_assignment_config`)
- **При обнаружении дефекта**: увеличивается на `increase_on_defect_percentage` (по умолчанию +2%)
- **Максимум**: 100% (полная проверка)
- Процент настраивается **отдельно для каждой фабрики и каждого этапа**

## 11. Отчёты по качеству

### 11.1 Отчёты по поставщикам

**GET** `/api/defects/supplier-reports`

Агрегированная статистика дефектов по типу глазури (прокси для поставщика):
- Общее количество записей и единиц продукции
- Количество и процент дефектов
- Фильтры: фабрика, период

### 11.2 Детальный отчёт по поставщику

**POST** `/api/defects/supplier-reports/generate`

Генерирует подробный отчёт:
- Процент дефектов за период
- **Top-10 причин дефектов** (из quality_checks)
- Привязка к конкретному поставщику и фабрике

### 11.3 Surplus-отчёты

**GET** `/api/defects/surplus-dispositions` -- записи о распределении излишков
**GET** `/api/defects/surplus-summary` -- сводка: по типам (showroom / casters / mana), ожидающие решения PM

## 12. Уведомления

QC менеджер получает уведомления в следующих случаях:

| Событие | Тип | Кто отправляет |
|---------|-----|----------------|
| Назначена QC-задача (после обжига) | Task | Система (автоматически) |
| Превышен порог дефектов | Alert | Система (при `record_defect`) |
| Repair SLA просрочен | Alert | Система |

QC менеджер **отправляет** уведомления:
- PM и CEO -- при создании QM Block (критический дефект)
- PM -- при разрешении QM Block

## 14. Типичные сценарии работы

### Сценарий 1: Стандартная проверка после обжига

1. Открыть дашборд `/quality`
2. На карточке "Pending QC" видно количество ожидающих проверки
3. Перейти на вкладку **QC Queue**
4. Выбрать позицию
5. Осмотреть изделие, нажать **OK** или **Defect**
6. При дефекте -- выбрать причину, добавить заметку, прикрепить фото
7. Submit

### Сценарий 2: Критический дефект -- блокировка

1. При инспекции обнаружен критический дефект
2. Нажать **Defect**, выбрать причину
3. Система автоматически создаёт QM Block и уведомляет PM
4. Позиция блокируется до разрешения
5. После исправления -- перейти на вкладку **QM Blocks**, нажать **Resolve**, ввести заметку

### Сценарий 3: Проблема на участке

1. Обнаружена системная проблема (например, загрязнение глазури)
2. Перейти на вкладку **Problem Cards**
3. Нажать **New Problem Card**
4. Описать проблему, указать участок
5. Отслеживать статус до закрытия

### Сценарий 4: AI-анализ фото дефекта

1. Сфотографировать дефектное изделие
2. Загрузить через `/api/quality/analyze-photo` с типом `quality`
3. Получить автоматический анализ: тип дефекта, уровень уверенности, описание
4. Использовать результат для заполнения инспекции

## 15. Связанные роли

| Роль | Взаимодействие с QC |
|------|---------------------|
| **Production Manager** | Получает уведомления о блоках, принимает решения по ремонту/списанию |
| **CEO** | Получает уведомления о критических блоках, может переопределить коэффициенты |
| **Owner** | Полный доступ, переопределение коэффициентов дефектов |
| **Sorter/Packer** | Выполняет физическую сортировку, сплит позиций |
| **Administrator** | Полный доступ к QC-функциям |
