## 1. Обзор роли и зоны ответственности

Production Manager (PM) -- это ключевая роль в Moonjar PMS. PM отвечает за весь производственный цикл от получения заказа до отгрузки готовой продукции.

### Уровень доступа

PM входит в группу `require_management` (наравне с Owner, Administrator, CEO) и имеет доступ ко всем производственным операциям:

| Возможность | PM | CEO | Admin | Owner |
|---|:---:|:---:|:---:|:---:|
| Создание/редактирование заказов | + | + | + | + |
| Управление позициями и статусами | + | + | + | + |
| Формирование батчей и загрузка печей | + | + | + | + |
| Force-unblock заблокированных позиций | + | + | + | + |
| Управление печами (CRUD) | + | + | + | + |
| Перепланирование расписания | + | + | + | + |
| Создание/редактирование задач | + | + | + | + |
| Управление правилами расхода | + | - | + | + |
| Одобрение запросов на отмену/изменение | + | + | + | + |
| Администрирование пользователей | - | - | + | + |

### Зоны ответственности PM

- **Производственное планирование**: приоритезация заказов, формирование очереди глазуровки и обжига
- **Контроль потока**: мониторинг статусов позиций по всем этапам (glazing -> firing -> sorting -> QC -> shipment)
- **Управление ограничениями**: печи являются главным ограничением (constraint) в TOC/DBR; PM управляет загрузкой печей
- **Разрешение блокировок**: нехватка материалов, ожидание рецепта/трафарета/подбора цвета, блокировки QM
- **Координация команды**: назначение задач сортировщикам, складу, закупщику

## 2. Вход в систему и навигация

### Авторизация

1. Откройте dashboard по адресу вашей инсталляции
2. Войдите через Google OAuth или email/password
3. После входа система определяет вашу роль и перенаправляет на `/manager`

### Основная навигация

PM имеет доступ к следующим страницам:

| Страница | URL | Назначение |
|---|---|---|
| Дашборд PM | `/manager` | Главная панель управления |
| Расписание | `/manager/schedule` | Очереди по секциям (глазуровка, обжиг, сортировка, QC) |
| Печи | `/manager/kilns` | Управление печами, загрузка, обслуживание |
| Материалы | `/manager/materials` | Остатки, транзакции, запросы на закупку |
| Правила расхода | `/consumption-rules` | Нормы расхода глазури/ангоба на м2 |
| Детали заказа | `/orders/:id` | Подробный просмотр заказа и его позиций |

> **Примечание**: Через Dashboard Constructor (настраивается Admin) PM может получить дополнительный доступ к дашбордам других ролей (например, CEO Dashboard в режиме чтения).

## 3. Дашборд PM: вкладки и виджеты

Главный дашборд PM (`/manager`, компонент `ManagerDashboard`) содержит следующие вкладки:

### 3.1. Orders (Заказы)

Основная вкладка. Показывает список всех производственных заказов с фильтрами.

- **Подвкладки**: Current / Archive
- **Фильтры**: поиск по номеру/клиенту, фильтр по статусу (New, In Production, Partially Ready, Ready for Shipment, Cancelled)
- **Действия**: создание нового заказа, загрузка заказа из PDF, переход в детали заказа

<!-- Screenshot: PM Dashboard - Orders tab -->
`[Скриншот: Вкладка Orders на дашборде PM]`

### 3.2. Tasks (Задачи)

Список задач, назначенных PM или его команде.

- Фильтры по статусу, типу, исполнителю
- Блокирующие задачи выделяются визуально

### 3.3. Materials (Материалы)

Сводка по материалам: low-stock алерты, запросы на закупку.

- **Low Stock**: материалы с балансом ниже минимума
- **Purchase Requests**: запросы на закупку, созданные PM или системой
- **Consumption Adjustments**: расхождения фактического и расчетного расхода (PM одобряет/отклоняет)

### 3.4. Defects (Дефекты)

Статистика дефектов и алерты по аномальным отклонениям.

- Баннер `DefectAlertBanner` -- уведомление при высоком проценте брака
- Баннер `AnomalyAlertBanner` -- алерт при аномальном расходе материалов

### 3.5. TPS (Toyota Production System)

Параметры производственной системы для мониторинга эффективности.

### 3.6. TOC (Theory of Constraints)

Визуализация буферных зон по методологии TOC/DBR:
- Зеленая зона -- работа идет по плану
- Желтая зона -- буфер начинает сокращаться
- Красная зона -- критическое отставание от плана

Компонент `TOCZonesTab` с визуализацией `BottleneckVisualization`.

### 3.7. Stone (Камень)

Вкладка `StoneReservationTab` -- резервирование каменных заготовок под позиции.

### 3.8. Kilns (Печи)

Краткая сводка по статусам печей прямо с дашборда.

### 3.9. AI Chat

Встроенный AI-ассистент для быстрого получения ответов по производству.

### Динамические вкладки

При наличии данных дашборд автоматически добавляет дополнительные вкладки:
- **Blocking** -- заблокированные позиции, требующие решения PM
- **Cancellations** -- запросы на отмену от Sales
- **Change Requests** -- запросы на изменение параметров заказа
- **Mismatch** -- позиции с расхождением цвета, требующие решения

## 4. Управление заказами

### 4.1. Просмотр заказов

**Путь**: Дашборд PM -> вкладка Orders

Заказы отображаются в виде таблицы с основной информацией:
- Номер заказа, клиент, источник (sales_webhook / pdf_upload / manual)
- Статус заказа
- Дедлайн
- Количество позиций

**Фильтры**:
- Поиск по номеру заказа или имени клиента
- Фильтр по статусу
- Фильтр по фабрике (Factory Selector в шапке)

### 4.2. Создание заказа вручную

1. Нажмите кнопку **"Create Order"** на вкладке Orders
2. Заполните данные заказа:
   - **Order Number** -- уникальный номер
   - **Client** -- имя клиента
   - **Factory** -- выберите фабрику (Bali / Java)
   - **Deadline** -- конечный дедлайн
   - **Items** -- добавьте позиции заказа с параметрами:
     - Color, Size, Application, Finishing
     - Quantity (шт.), Thickness (мм, по умолчанию 11)
     - Collection, Product Type (tile/sink/pebble...)
     - Shape (rectangle, round, triangle, octagon, freeform)
3. Нажмите **"Create"**

> Система автоматически создаст `OrderPosition` для каждого item и назначит статус `PLANNED`.

### 4.3. Загрузка заказа из PDF

1. Нажмите **"Upload PDF"**
2. Загрузите PDF-файл с заказом
3. Система распознает данные через AI-парсер
4. Проверьте и подтвердите распознанные данные (`/orders/confirm-pdf`)

### 4.4. Статусы заказов

| Статус | Описание |
|---|---|
| `new` | Все позиции в статусе PLANNED |
| `in_production` | Хотя бы одна позиция в работе |
| `partially_ready` | Часть позиций готова (packed/quality_check_done/ready_for_shipment) |
| `ready_for_shipment` | Все позиции готовы к отгрузке |
| `shipped` | Все позиции отгружены |
| `cancelled` | Все позиции отменены |

> Статус заказа рассчитывается автоматически на основе статусов его позиций. PM может установить `status_override` для ручного управления.

### 4.5. Отгрузка заказа

1. Откройте заказ -> нажмите **"Ship Order"**
2. Система переведет все позиции в статус `SHIPPED`
3. Если заказ создан через Sales webhook, уведомление автоматически отправится обратно в Sales

### 4.6. Удаление заказов

PM может удалять заказы, если эта возможность включена на уровне фабрики (параметр `pm_can_delete_orders`). Удаление необратимо -- удаляются все позиции и связанные задачи.

## 5. Управление позициями

### 5.1. Жизненный цикл позиции

Каждый OrderItem при создании заказа превращается в `OrderPosition`. Позиция проходит через следующие статусы:

```
PLANNED
  |
  +-- INSUFFICIENT_MATERIALS (блокировка: нехватка материалов)
  +-- AWAITING_RECIPE (блокировка: нет рецепта)
  +-- AWAITING_STENCIL_SILKSCREEN (блокировка: нет трафарета)
  +-- AWAITING_COLOR_MATCHING (блокировка: подбор цвета)
  +-- AWAITING_SIZE_CONFIRMATION (блокировка: уточнение размера)
  |
  v
SENT_TO_GLAZING -> ENGOBE_APPLIED -> ENGOBE_CHECK -> GLAZED
  |
  v
PRE_KILN_CHECK -> LOADED_IN_KILN -> FIRED
  |
  +-- REFIRE (повторный обжиг)
  +-- AWAITING_REGLAZE (повторная глазуровка)
  |
  v
TRANSFERRED_TO_SORTING -> PACKED
  |
  v
SENT_TO_QUALITY_CHECK -> QUALITY_CHECK_DONE
  |
  +-- BLOCKED_BY_QM (блокировка QM)
  |
  v
READY_FOR_SHIPMENT -> SHIPPED
```

### 5.2. Смена статуса позиции

1. Откройте заказ (страница деталей или расписание)
2. Найдите нужную позицию
3. Используйте `StatusDropdown` для выбора нового статуса
4. Система проверит допустимый переход через `/positions/{id}/allowed-transitions`
5. При смене статуса:
   - Автоматически пересчитается статус заказа
   - Запустится перепланирование (reschedule) позиции
   - Для статусов глазуровки -- произойдет резервирование/списание материалов

### 5.3. Нумерация и метки позиций

Каждая позиция получает номер внутри заказа: `#1`, `#2`, `#3` и т.д.

При сплите (разделении) позиции создаются подпозиции: `#1.1`, `#1.2`, `#1.3`.

### 5.4. Сплит позиций

PM может разделить позицию на несколько частей:

**Production Split** (`/positions/{id}/split-production`):
- Разделение позиции для разных печей или этапов
- Указывается количество (qty) для отделяемой части
- Создается дочерняя позиция с `split_index`

**Sorting Split** (`/positions/{id}/split`):
- Происходит на этапе сортировки
- Разделение по категориям: A-sort, B-sort, C-sort, showroom, repair, grinding, utilization

### 5.5. Merge (объединение) позиций

Разделенные подпозиции можно объединить обратно:

1. Проверьте объединяемые позиции: `/positions/{id}/mergeable-children`
2. Выполните merge: `/positions/{id}/merge`

### 5.6. Переназначение батча

PM может переместить позицию из одного батча в другой:

1. Выберите позицию
2. Используйте `/positions/{id}/reassign-batch`
3. Укажите новый `batch_id`

### 5.7. Приоритезация позиций

PM может менять приоритет позиций через drag-and-drop (reorder):

1. Перетащите позиции в нужном порядке
2. Система обновит `priority_order` для каждой позиции
3. Автоматически запустится перепланирование

Приоритет влияет на порядок в расписании: `priority_order -> deadline -> position_number -> split_index`.

### 5.8. Просмотр наличия камня

Для каждой позиции можно проверить доступность каменных заготовок:

`GET /positions/{id}/stock-availability`

Возвращает: доступное количество, зарезервированное количество, дефицит.

## 6. Материалы и складские остатки

### 6.1. Страница материалов

**Путь**: `/manager/materials`

Страница показывает полный каталог материалов с остатками для выбранной фабрики.

**Типы материалов**:
- `stone` -- камень
- `pigment` -- пигменты
- `frit` -- фритты
- `oxide_carbonate` -- оксиды и карбонаты
- `other_bulk` -- прочее сыпучее
- `packaging` -- упаковка
- `consumable` -- расходные материалы

**Фильтры**: по типу, подгруппе, группе, секции склада, low stock, поиск по названию.

### 6.2. Low Stock алерты

`GET /materials/low-stock` -- материалы с балансом ниже минимального порога.

На дашборде PM в таблице low stock отображается:
- Название материала
- Текущий остаток
- Минимальный остаток
- Дефицит (min_balance - balance)

### 6.3. Создание материала

1. Нажмите **"Add Material"**
2. Заполните:
   - Название
   - Тип (stone, pigment, frit...)
   - Подгруппа
   - Единица измерения (kg, g, L, ml, pcs, m, m2)
   - Начальный остаток
   - Минимальный остаток
   - Поставщик
   - Секция склада
3. При создании без указания фабрики -- stock создается для ВСЕХ активных фабрик

### 6.4. Транзакции

PM может создавать транзакции:

- **receive** -- приход материала на склад
- **manual_write_off** -- ручное списание

Автоматические транзакции:
- **reserve** -- резервирование под позицию (при переходе в glazing)
- **unreserve** -- снятие резерва (при отмене или force-unblock)
- **consume** -- фактическое списание при глазуровке

### 6.5. Запросы на закупку

PM может создавать запросы на закупку:

1. Нажмите **"Create Purchase Request"**
2. Укажите: фабрику, поставщика, перечень материалов с количествами, примечания
3. Запрос появится у Purchaser в его дашборде

### 6.6. Корректировки расхода (Consumption Adjustments)

Когда фактический расход материала отличается от расчетного, система создает `ConsumptionAdjustment`.

PM должен рассмотреть и одобрить или отклонить каждую корректировку:
- **Approve** (`/materials/consumption-adjustments/{id}/approve`) -- подтвердить фактический расход
- **Reject** (`/materials/consumption-adjustments/{id}/reject`) -- отклонить, оставить расчетный

### 6.7. Эффективный остаток

`GET /materials/effective-balance` -- показывает баланс с учетом всех резервов.

Формула: `effective = balance - (sum(reserve) - sum(unreserve))`

## 7. Расписание производства

### 7.1. Страница расписания

**Путь**: `/manager/schedule`

Расписание разделено на секции (вкладки):

| Секция | Статусы позиций |
|---|---|
| **Glazing** | planned, insufficient_materials, awaiting_recipe, awaiting_stencil_silkscreen, awaiting_color_matching, engobe_applied, engobe_check, glazed, pre_kiln_check, sent_to_glazing |
| **Firing** | loaded_in_kiln, fired, refire, awaiting_reglaze |
| **Sorting** | transferred_to_sorting, packed, ready_for_shipment |
| **QC** | sent_to_quality_check, quality_check_done, blocked_by_qm |
| **Kilns** | Батчи по печам (planned / in_progress) |

### 7.2. Glazing Schedule

Показывает все позиции в очереди на глазуровку. Порядок:
1. Ручной приоритет (`priority_order`)
2. Дедлайн заказа (ближайший -- первый)
3. Номер позиции
4. Индекс сплита

PM может:
- Менять статус позиции через StatusDropdown
- Перетаскивать позиции для изменения приоритета (reorder)
- Удалять позиции (если разрешено настройками фабрики)

### 7.3. Firing Schedule

Позиции, загруженные в печь или уже обожженные. Привязаны к батчам.

### 7.4. Kiln Schedule

Визуализация загрузки печей:
- Каждая печь показывает свои активные батчи (planned / in_progress)
- Видно количество позиций и общую площадь

### 7.5. Auto-Form Batches

Кнопка **"Auto-form Batches"** на вкладке Kilns:

1. Выберите конкретную фабрику
2. Нажмите "Auto-form Batches"
3. Система автоматически:
   - Соберет позиции, готовые к обжигу (pre_kiln_check, glazed)
   - Сгруппирует по температуре обжига
   - Назначит на подходящие печи
   - Создаст батчи

### 7.6. Перепланирование

PM может инициировать полное перепланирование:

- **Перепланирование заказа**: `POST /schedule/orders/{order_id}/reschedule`
- **Перепланирование фабрики**: `POST /schedule/factory/{factory_id}/reschedule`

Перепланирование пересчитывает плановые даты для всех позиций:
- `planned_glazing_date` -- плановая дата глазуровки
- `planned_kiln_date` -- плановая дата обжига
- `planned_sorting_date` -- плановая дата сортировки
- `planned_completion_date` -- плановая дата завершения
- `estimated_kiln_id` -- предполагаемая печь

### 7.7. Просмотр расписания заказа

`GET /schedule/orders/{order_id}/schedule` -- полное расписание заказа для передачи Sales.

## 8. Управление печами (Kilns)

### 8.1. Страница печей

**Путь**: `/manager/kilns`

Показывает все печи с подробными характеристиками.

### 8.2. Типы печей

| Тип | Описание |
|---|---|
| `big` | Большая печь |
| `small` | Маленькая печь |
| `raku` | Печь Раку (60x100 см) |

### 8.3. Создание печи

1. Нажмите **"Add Kiln"**
2. Заполните:
   - Название
   - Фабрика
   - Тип (big/small/raku)
   - Размеры (width x depth x height) в см
   - Рабочая площадь
   - Многоуровневая (да/нет)
   - Коэффициент загрузки (по умолчанию 0.8)
   - Емкость (м2 и шт.)
   - Термопара (chinese / indonesia_manufacture)
   - Контрольный кабель
   - Устройство управления (oven / moonjar)

### 8.4. Статусы печей

| Статус | Описание |
|---|---|
| `active` | Рабочая, доступна для загрузки |
| `maintenance_planned` | Плановое ТО запланировано |
| `maintenance_emergency` | Аварийный ремонт |
| `inactive` | Неактивна |

### 8.5. Аварийная поломка (Breakdown)

При поломке печи:

1. Откройте печь -> нажмите **"Report Breakdown"**
2. Укажите причину поломки и ориентировочное время ремонта
3. Система автоматически:
   - Переведет печь в статус `maintenance_emergency`
   - Переназначит батчи на другие печи
   - Создаст запись обслуживания
   - Уведомит PM и CEO через Telegram
   - Отправит сообщение в групповой чат мастеров

### 8.6. Восстановление печи

1. После ремонта нажмите **"Restore Kiln"**
2. Печь вернется в статус `active`
3. Открытые записи обслуживания будут завершены

### 8.7. Правила загрузки (Loading Rules)

Каждая печь может иметь индивидуальные правила загрузки (JSONB):
- Нажмите **"Loading Rules"** на карточке печи
- Настройте правила по коллекциям и типам продукции

### 8.8. Правила ротации (Rotation Rules)

Правила чередования типов глазури в печи для предотвращения контаминации:

- `glaze_sequence` -- порядок глазурей
- `cooldown_minutes` -- время охлаждения между типами
- `incompatible_pairs` -- несовместимые пары глазурей

PM может:
- Просмотреть правила: `GET /kilns/{id}/rotation-rules`
- Создать/обновить: `PUT /kilns/{id}/rotation-rules`
- Проверить совместимость: `GET /kilns/{id}/rotation-check?proposed_glaze=...`

### 8.9. Обслуживание печей (Maintenance)

PM управляет расписанием обслуживания:

1. Просмотр предстоящего ТО: `GET /kilns/maintenance/upcoming?days=30`
2. Создание записи ТО на конкретную печь: кнопка **"Schedule Maintenance"**
3. Параметры:
   - Тип обслуживания (из каталога KilnMaintenanceType)
   - Дата и время
   - Ориентировочная продолжительность
   - Требуется ли пустая печь / остывшая печь / отключение питания
   - Периодичность (recurring) с интервалом в днях
4. Завершение ТО: **"Complete"** -- если ТО периодическое, автоматически создается следующая запись

### 8.10. Константы печей (Kiln Constants)

На странице `/manager/kilns` доступна таблица `KilnConstantsTable`:
- Двойной режим: ручной + производственный
- Формула: `max(calculated, actual)`

### 8.11. Предпросмотр загрузки

Перед назначением позиции в печь можно выполнить предпросмотр:

`POST /batches/capacity-preview` с `position_id` и `kiln_id`

Возвращает:
- Оптимальный и альтернативный способы загрузки (flat/edge)
- Штук на уровень, количество уровней, общее количество
- Утилизация площади
- Информация о заполнителе (filler tiles для маленьких печей)

## 9. Батчи: формирование и жизненный цикл

### 9.1. Что такое батч

Батч (Batch) -- это набор позиций, назначенных для обжига в одной печи в определенную дату.

### 9.2. Жизненный цикл батча

```
SUGGESTED -> PLANNED -> IN_PROGRESS -> DONE
                |
                +-- (rejected -> deleted)
```

| Статус | Описание |
|---|---|
| `suggested` | Предложен системой (auto-form в режиме suggest) |
| `planned` | Подтвержден PM, запланирован к обжигу |
| `in_progress` | Печь загружена, обжиг идет |
| `done` | Обжиг завершен |

### 9.3. Автоматическое формирование

`POST /batches/auto-form`:
- **mode: "auto"** -- система создает батчи сразу в статусе PLANNED
- **mode: "suggest"** -- система предлагает батчи (SUGGESTED), PM подтверждает/отклоняет

### 9.4. Ручное создание батча

Через страницу Schedule, вкладка Kilns:

1. Выберите печь и дату
2. Выберите позиции из очереди
3. Создайте батч

### 9.5. Подтверждение/отклонение предложенного батча

**Confirm** (`POST /batches/{id}/confirm`):
- Можно изменить состав позиций (добавить/убрать)
- Можно изменить дату батча
- Добавить примечания

**Reject** (`POST /batches/{id}/reject`):
- Батч удаляется
- Позиции возвращаются в очередь

### 9.6. Запуск и завершение обжига

**Start** (`POST /batches/{id}/start`):
- Батч -> IN_PROGRESS
- Все позиции -> LOADED_IN_KILN

**Complete** (`POST /batches/{id}/complete`):
- Батч -> DONE
- Все позиции -> FIRED и далее автоматически маршрутизируются

### 9.7. Фото обжига

После выгрузки из печи можно загрузить фотографии:
- `POST /batches/{id}/photos` -- добавить фото
- `GET /batches/{id}/photos` -- просмотр фотографий батча

## 10. Блокирующие задачи и Force-Unblock

### 10.1. Типы блокировок

Позиция может быть заблокирована по следующим причинам:

| Статус блокировки | Причина | Как разрешить |
|---|---|---|
| `insufficient_materials` | Не хватает материалов для глазуровки | Пополнить склад ИЛИ force-unblock |
| `awaiting_recipe` | Не привязан рецепт | Привязать рецепт к позиции ИЛИ force-unblock |
| `awaiting_stencil_silkscreen` | Нет трафарета/шелкографии | Заказать трафарет, дождаться ИЛИ force-unblock |
| `awaiting_color_matching` | Подбор цвета не завершен | Завершить подбор ИЛИ force-unblock |
| `awaiting_size_confirmation` | Размер не подтвержден | Подтвердить размер ИЛИ force-unblock |
| `blocked_by_qm` | Заблокирован менеджером качества | Решить проблему с QM ИЛИ force-unblock |

### 10.2. Blocking Summary

**Вкладка Blocking** на дашборде PM показывает:
- Общее количество заблокированных позиций
- Группировку по типам блокировки
- Для каждой позиции:
  - Номер заказа, позиция, цвет, размер, количество
  - Связанные блокирующие задачи
  - Дефициты материалов (для insufficient_materials)
  - Время нахождения в блокировке

### 10.3. Force-Unblock

PM может принудительно разблокировать позицию, когда стандартное решение невозможно или нецелесообразно.

**Пошаговая инструкция**:

1. Найдите заблокированную позицию (вкладка Blocking или в расписании)
2. Нажмите **"Force Unblock"**
3. **Обязательно** укажите причину в поле Notes (для аудита)
4. Подтвердите действие

**Что происходит при Force-Unblock**:

- `insufficient_materials` -> выполняется принудительное резервирование (баланс материала может уйти в минус!), позиция -> PLANNED
- `awaiting_recipe` -> позиция -> PLANNED (PM берет на себя ответственность)
- `awaiting_stencil_silkscreen` / `awaiting_color_matching` -> блокирующие задачи закрываются, позиция -> PLANNED
- `blocked_by_qm` -> блокирующие задачи QM закрываются, позиция -> PLANNED

> **Важно**: Force-Unblock создает запись аудита (Task типа `mana_confirmation`) с информацией о том, кто, когда и почему выполнил разблокировку. Все отрицательные балансы фиксируются в `negative_balances`.

### 10.4. Resolve Color Mismatch

Специальный случай блокировки -- расхождение цвета:

`POST /positions/{id}/resolve-color-mismatch`

PM решает, что делать с позицией, цвет которой не совпадает с ожидаемым.

## 11. TOC/DBR: буферные зоны

### 11.1. Принцип работы

Moonjar PMS использует методологию Theory of Constraints (TOC) с планированием Drum-Buffer-Rope (DBR), где печи -- основное ограничение (drum).

Для каждого заказа система рассчитывает:
- **Buffer penetration** = (прошедшее время / общее время) * 100%
- **Work completion** = (завершенные позиции / всего позиций) * 100%
- **Delta** = work_completion - buffer_penetration

### 11.2. Зоны

| Зона | Условие | Действие PM |
|---|---|---|
| Зеленая | delta >= -5% | Все в порядке, штатная работа |
| Желтая | -20% <= delta < -5% | Обратить внимание, повысить приоритет |
| Красная | delta < -20% или дедлайн прошел | Срочные меры: повышение приоритета, дополнительные ресурсы |

### 11.3. Как использовать

1. Откройте вкладку **TOC** на дашборде PM
2. Просмотрите заказы по зонам
3. Красные заказы требуют немедленного внимания:
   - Проверьте, нет ли блокировок
   - Повысьте приоритет позиций
   - Рассмотрите возможность дополнительного обжига

## 12. Правила расхода (Consumption Rules)

### 12.1. Назначение

Consumption Rules определяют нормы расхода глазури и ангоба (мл на м2) для автоматического расчета потребности в материалах.

**Путь**: `/consumption-rules`

**Доступ**: PM и Admin (`require_admin_or_pm`)

### 12.2. Параметры правила

| Параметр | Описание |
|---|---|
| `rule_number` | Порядковый номер (для сортировки) |

| `collection` | Коллекция (фильтр) |
| `color_collection` | Цветовая коллекция |
| `product_type` | Тип продукта (tile, sink...) |
| `size_id` | Размер |
| `shape` | Форма |
| `thickness_mm_min/max` | Диапазон толщины |
| `place_of_application` | Место применения |
| `recipe_type` | Тип рецепта (engobe, glaze...) |
| `application_method` | Метод нанесения |
| `consumption_ml_per_sqm` | Расход мл на м2 |
| `coats` | Количество слоев |
| `specific_gravity_override` | Переопределение удельного веса |
| `priority` | Приоритет правила (при совпадении нескольких) |

### 12.3. Управление правилами

## 16. Запросы на отмену и изменение заказов

### 16.1. Запросы на отмену (Cancellation Requests)

Когда Sales отправляет запрос на отмену заказа, PM принимает решение:

**Принять** (`POST /orders/{id}/accept-cancellation`):
- Заказ отменяется
- Все позиции -> CANCELLED
- Резервы материалов освобождаются

**Отклонить** (`POST /orders/{id}/reject-cancellation`):
- Заказ продолжает производство
- Sales получает уведомление об отклонении

### 16.2. Запросы на изменение (Change Requests)

Sales может запросить изменение параметров заказа (количество, цвет, размер и т.д.):

**Одобрить** (`POST /orders/{id}/approve-change`):
- Изменения применяются к заказу
- Позиции обновляются
- Запускается перепланирование

**Отклонить** (`POST /orders/{id}/reject-change`):
- Изменения не применяются
- Sales получает уведомление

Эти запросы появляются как отдельные вкладки **Cancellations** и **Change Requests** на дашборде PM.

## 17. Ежедневные рутины PM

### Утро (начало смены)

1. **Проверить дашборд** -- вкладка Orders: есть ли новые заказы?
2. **Проверить блокировки** -- вкладка Blocking: сколько позиций заблокировано? Есть ли критические?
3. **Проверить TOC зоны** -- вкладка TOC: есть ли заказы в красной зоне?
4. **Проверить low stock** -- вкладка Materials: нет ли критических дефицитов?
5. **Проверить задачи** -- вкладка Tasks: есть ли просроченные задачи?

### В течение дня

6. **Расписание глазуровки** -- `/manager/schedule`, вкладка Glazing: проверить очередь, при необходимости перестроить приоритеты
7. **Формирование батчей** -- вкладка Kilns: сформировать батчи для обжига, запустить обжиг
8. **Решение блокировок** -- force-unblock или организация решения (заказ трафарета, пополнение склада и т.д.)
9. **Мониторинг обжига** -- проверить статусы IN_PROGRESS батчей, завершить готовые
10. **Одобрение корректировок** -- Consumption Adjustments: одобрить/отклонить расхождения расхода

### Вечер (конец смены)

11. **Проверить completion** -- сколько позиций завершено за день?
12. **Подготовить батчи на завтра** -- сформировать батчи на следующий день
13. **Проверить запросы Sales** -- Cancellations / Change Requests: принять решения

### Еженедельно

14. **Обслуживание печей** -- проверить расписание ТО: `/kilns/maintenance/upcoming?days=7`
15. **Правила расхода** -- пересмотреть consumption rules при необходимости
16. **Перепланирование фабрики** -- при серьезных изменениях запустить full reschedule
