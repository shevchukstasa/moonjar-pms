# Руководство сортировщика/упаковщика (Sorter/Packer)

> Moonjar PMS -- система управления производством каменных изделий.
> Данный документ описывает роль **sorter_packer** и все рабочие процессы, доступные этой роли.

---

## 1. Зона ответственности

Сортировщик/упаковщик -- это сотрудник, который:

1. **Принимает обожжённую продукцию с печи** (позиции в статусе `transferred_to_sorting`)
2. **Осматривает каждую плитку** и распределяет по категориям (годная, дожиг, ремонт, несовпадение цвета, шлифовка, списание)
3. **Упаковывает годную продукцию** и отправляет на проверку качества (QC)
4. **Работает со шлифовальным запасом** -- принимает решения по позициям на шлифовке
5. **Фотографирует упаковку** -- загружает фото упакованных позиций в систему
6. **Выполняет задачи** -- получает и закрывает назначенные задания (отправка в шоурум, фотосъёмка и т.д.)

### Права доступа

Роль `sorter_packer` входит в группу `require_sorting`, что даёт доступ к:
- Просмотр позиций и заказов (общий доступ)
- Операция **Split** (разбивка позиции после обжига)
- Загрузка и удаление **фото упаковки**
- Изменение статуса позиций в рамках сортировки
- Выполнение назначенных задач

Сортировщик **НЕ** имеет доступа к:
- Управлению пользователями, фабриками, настройками
- Финансовым данным
- Созданию/редактированию заказов
- Панелям CEO, Owner, PM, Admin

## 2. Дашборд сортировщика

При входе в систему сортировщик попадает на страницу **Sorting & Packing**.

### 2.1. KPI-карточки (верхняя панель)

Три ключевых показателя всегда видны:

| Карточка | Описание |
|---|---|
| **Awaiting Sorting** | Количество позиций, ожидающих сортировки (статус `transferred_to_sorting`) |
| **Packed** | Количество упакованных позиций (статус `packed`) |
| **Open Tasks** | Количество открытых задач, назначенных сортировщику |

### 2.2. Вкладки

Дашборд содержит **5 вкладок**:

| Вкладка | Назначение |
|---|---|
| **Sorting** | Приём и разбивка обожжённых позиций |
| **Packing** | Отправка упакованных позиций на QC |
| **Grinding** | Работа с позициями на шлифовке |
| **Photos** | Загрузка и просмотр фото упаковки |
| **Tasks** | Список назначенных задач |

## 3. Процесс сортировки (вкладка Sorting)

### 3.1. Общий поток

```
Печь (обжиг) --> transferred_to_sorting --> СОРТИРОВКА --> packed / sub-positions
```

### 3.2. Пошаговая инструкция

1. **Открыть вкладку Sorting** -- отображается список позиций со статусом `transferred_to_sorting`
2. **Выбрать позицию** -- нажать на карточку позиции (номер заказа, цвет, размер, количество)
3. **Для стоковых позиций** (коллекция "Сток" / "Stock") -- автоматически показывается панель **Stock Availability**:
   - Сколько нужно по заказу
   - Сколько есть на текущей фабрике
   - Сколько на других фабриках
   - Достаточно ли всего запаса
4. **Заполнить форму Split** -- распределить общее количество плиток по категориям

### 3.3. Категории при разбивке (Split)

| Категория | Поле | Что происходит дальше |
|---|---|---|
| **Good** (Годные) | `good_quantity` | Основная позиция переходит в статус `packed`. Автоматически списываются упаковочные материалы (коробки, прокладки) |
| **Refire** (Дожиг) | `refire_quantity` | Создаётся суб-позиция со статусом `refire` -- плитка уже глазурована, нужен повторный обжиг |
| **Repair** (Ремонт) | `repair_quantity` | Суб-позиция со статусом `sent_to_glazing` -- нужна переглазуровка перед повторным обжигом. Добавляется в очередь ремонта (Repair Queue) для отслеживания SLA |
| **Color Mismatch** (Несовпадение цвета) | `color_mismatch_quantity` | Суб-позиция со статусом `planned` -- перезапуск с начала производства |
| **Grinding** (Шлифовка) | `grinding_quantity` | Добавляется в Grinding Stock (запас на шлифовку) |
| **Write-off** (Списание) | `write_off_quantity` | Создаётся запись о дефекте (DefectRecord), плитки списываются |

### 3.4. Правила валидации

- **Сумма всех категорий ОБЯЗАНА равняться общему количеству позиции**
- Система показывает индикатор: `Total: 47 / 47` (зелёный, если совпадает; красный, если нет)
- Кнопка **Submit Split** заблокирована, пока сумма не сойдётся
- Поле Notes (заметки) -- необязательное

### 3.5. Что происходит после Submit

1. Основная позиция обновляется: количество = `good_quantity`, статус = `packed`
2. Для каждой категории с количеством > 0 создаются суб-позиции
3. Суб-позиции наследуют от родительской: цвет, размер, коллекцию, рецепт, тип продукта
4. При наличии излишков (good > заказанного) -- срабатывает автоматическое распределение:
   - **10x10, базовый цвет** --> Шоурум (создаётся задача на отправку)
   - **10x10, небазовый цвет** --> Коробка для подставок (Coaster box)
   - **Другие размеры** --> Mana (склад дефектной продукции)

## 4. Процесс упаковки (вкладка Packing)

### 4.1. Общий поток

```
packed --> [Сортировщик нажимает "Send to QC"] --> sent_to_quality_check
```

### 4.2. Пошаговая инструкция

1. На вкладке **Packing** отображаются все позиции со статусом `packed`
2. Для каждой позиции доступна кнопка **Send to QC**
3. При нажатии появляется диалог подтверждения: "This position will be sent for quality inspection. Continue?"
4. После подтверждения статус меняется на `sent_to_quality_check`
5. Далее менеджер по качеству проводит инспекцию

### 4.3. Типы упаковочных коробок

Система поддерживает настраиваемые типы коробок (управляются администратором):
- Каждый тип коробки привязан к материалу
- Для каждого размера плитки задаётся вместимость (штук или кв.м. на коробку)
- Правила прокладок (spacers): какой материал-прокладка и сколько штук на коробку

При Split автоматически списываются упаковочные материалы со склада.

## 5. Работа с дефектами

### 5.1. Классификация дефектов при сортировке

Система различает следующие типы дефектов:

| Тип дефекта | Маршрут | Конечный результат |
|---|---|---|
| **crack** (трещина) | --> Mana (списание) | Позиция отменяется (`cancelled`). Записывается в ManaShipment |
| **stuck** (прилипание) | --> Mana (списание) | Аналогично crack |
| **color_mismatch** (несовпадение цвета) | --> Ожидание решения PM | Создаётся блокирующая задача для менеджера: перекрасить / списать / Mana |
| **glaze_defect** (дефект глазури) | --> Ремонт (переглазуровка) | Позиция отправляется на глазуровку (`sent_to_glazing`). Добавляется в Repair Queue |
| **shape_defect** (дефект формы) | --> Ремонт (шлифовка + переглазуровка) | Аналогично glaze_defect, но требует предварительной шлифовки |
| **grinding** (шлифовка) | --> Grinding Stock | Добавляется в запас на шлифовку |

### 5.2. Маршрут Mana

**Mana** -- это склад/пункт назначения для дефектной продукции, которую нельзя исправить.

- Списанные плитки накапливаются в ManaShipment (статус `pending`)
- Когда накопится **>= 100 штук** -- автоматически создаётся задача для менеджера производства на принятие решения об отправке
- Менеджер решает: отправить / подождать

### 5.3. Запись дефектов

Каждый дефект фиксируется как `DefectRecord` с полями:
- Фабрика
- Этап (`sorting`)
- Тип дефекта
- Количество
- Исход (outcome): `to_mana`, `repair`, `reglaze`, `grinding`, `write_off`

## 6. Шлифовка (вкладка Grinding)

### 6.1. Назначение

Вкладка Grinding показывает позиции со статусом `grinding` -- плитки, которые требуют шлифовки перед дальнейшей обработкой.

### 6.2. Доступные действия

Для каждой позиции сортировщик может выбрать одно из двух действий:

| Действие | Результат |
|---|---|
| **Grind** (Шлифовать) | Статус меняется на `awaiting_reglaze` -- после шлифовки плитка пойдёт на переглазуровку |
| **Mana** (Списать) | Статус меняется на `mana_confirmation` -- плитка отправляется в Mana (необратимо) |

Каждое действие требует подтверждения через диалоговое окно.

---

## 7. Фотографии упаковки (вкладка Photos)

### 7.1. Назначение

Фотофиксация упакованной продукции -- важная часть контроля качества. Фото прикрепляются к конкретным позициям.

### 7.2. Пошаговая инструкция

1. **Выбрать позицию** из выпадающего списка (отображаются позиции со статусами: `packed`, `transferred_to_sorting`, `sent_to_quality_check`)
2. **Загрузить фото**:
   - Нажать на область загрузки файла
   - Выбрать изображение (максимум **10 МБ**)
   - Добавить заметку (необязательно)
   - Нажать **Upload Photo**
3. **Просмотр галереи**: под формой загрузки отображается сетка всех загруженных фото для выбранной позиции
4. **Удаление фото**: кнопка Delete на каждом фото (удалить может только автор загрузки или руководство)

### 7.3. Хранение

Фотографии сохраняются в Supabase Storage (или локально как fallback). В базе данных хранится URL.

### 7.4. AI-анализ фото

Система поддерживает автоматический анализ фото через LLM (Claude Vision):
- **quality** -- обнаружение дефектов, трещин, проблем с цветом
- **packing** -- чтение данных этикетки (номер заказа, количество, размер)
- **scale** -- считывание веса, определение цвета пигмента

## 8. Задачи (вкладка Tasks)

### 8.1. Типы задач, назначаемых сортировщику

| Тип задачи | Описание |
|---|---|
| **showroom_transfer** | Отправить плитки в шоурум (излишки базовых цветов 10x10) |
| **photographing** | Отправить на фотосъёмку |
| **packing_photo** | Сделать фото упаковки |
| **quality_check** | Контроль качества |
| **stencil_order** | Заказ трафарета |
| **silk_screen_order** | Заказ шёлкотрафарета |
| **color_matching** | Подбор цвета |
| **material_order** | Заказ материала |

### 8.2. Работа с задачами

1. Список задач показывает: тип, описание, связанный заказ
2. **Блокирующие задачи** отмечены красным бейджем "Blocking" -- они останавливают производственный процесс и требуют приоритетного выполнения
3. Для завершения задачи нажать **Complete** и подтвердить в диалоге

## 9. Статусный цикл позиции (с точки зрения сортировщика)

```
transferred_to_sorting    <-- Позиция приходит с печи
       |
       v
   [SPLIT]
       |
       +---> packed                 (годные плитки)
       |        |
       |        v
       |   sent_to_quality_check    (отправлено на QC)
       |        |
       |        v
       |   quality_check_done       (QC пройден)
       |        |
       |        v
       |   ready_for_shipment       (готово к отгрузке)
       |
       +---> refire                 (дожиг)
       +---> sent_to_glazing        (ремонт/переглазуровка)
       +---> planned                (несовпадение цвета -- перезапуск)
       +---> grinding (IN_STOCK)    (шлифовка)
       +---> cancelled              (списание)
```

## 10. Учёт количества: план vs факт

### 10.1. Контроль при Split

- Система строго следит за тем, чтобы **сумма всех категорий = общему количеству позиции**
- Невозможно ввести больше или меньше плиток, чем было загружено в печь
- Каждая единица продукции должна быть отнесена к одной из 6 категорий

### 10.2. Отслеживание стока (Stock)

Для позиций коллекции "Сток" система показывает:
- Сколько нужно по заказу (`needed`)
- Сколько доступно на текущей фабрике (`factory_available`)
- Сколько доступно на других фабриках (с разбивкой)
- Общий итог по всем фабрикам (`total_available`)
- Статус: достаточно / недостаточно

Если запаса недостаточно -- после сортировки менеджер производства получает уведомление.

### 10.3. Автоматическое списание упаковки

При выполнении Split система автоматически:
1. Определяет тип коробки по размеру плитки
2. Рассчитывает количество коробок: `ceil(good_quantity / pieces_per_box)`
3. Рассчитывает количество прокладок
4. Списывает материалы со склада фабрики

## 11. Складское PWA (warehouse-pwa)

Для работы на складе существует отдельное приложение (PWA), доступное на порту 5175.

### Доступные страницы:

| Страница | Назначение |
|---|---|
| **Scan** | Сканирование штрих-кода или ввод кода позиции. Показывает информацию о позиции: заказ, продукт, размер, цвет, количество, статус, фабрика |
| **Receive** | Подтверждение приёмки материалов |
| **Inventory** | Просмотр складских остатков |

PWA интерфейс на **индонезийском языке** (Bahasa Indonesia) и оптимизирован для мобильных устройств.

## 12. Частые вопросы

**В: Что делать, если плитка не подходит ни под одну категорию дефекта?**
О: Используйте категорию Write-off (списание) и добавьте пояснение в поле Notes.

**В: Можно ли изменить результат сортировки после Submit?**
О: Нет, операция Split необратима. При ошибке обратитесь к менеджеру производства.

**В: Что означает "Blocking" на задаче?**
О: Блокирующая задача останавливает дальнейшее продвижение заказа. Выполняйте такие задачи в первую очередь.

**В: Куда деваются суб-позиции после ремонта?**
О: После успешного ремонта (переглазуровка + обжиг) суб-позиция возвращается на сортировку (`transferred_to_sorting`) и проходит весь цикл заново. При успехе она объединяется (merge) с родительской позицией.

**В: Как работает автоматическое списание упаковки?**
О: При Split система определяет тип коробки по размеру плитки и автоматически списывает нужное количество коробок и прокладок со склада фабрики. Если материалов не хватает -- операция Split всё равно пройдёт, но в логе появится предупреждение.
